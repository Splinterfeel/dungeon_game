# AGENTS.md

Инструкции для агентов (Claude Code и др.), работающих в этом репозитории.

## Что это за проект

Сервер пошаговой онлайн-игры на WebSocket. Игроки объединяются в две команды и
сражаются друг с другом (PvP), плюс на карте есть нейтральные враги под
управлением простого ИИ (PvE-элемент).

**Целевой сеттинг:** боевые роботы с пилотами внутри — идейно близко к
Front Mission 3, но онлайн, с двумя командами игроков вместо одиночной
кампании. Пилоты прокачивают статы, открывают новые детали, собирают
робота и тонко настраивают его сборку. **Главный фокус проекта —
кастомизация робота/лоадаута**, а не глубина боевой системы самой по себе.
Долгосрочная защита от «мёртвого онлайна» — боты: подлив в матчи (модель
PUBG) и асинхронное PvP против сборок других игроков — см. раздел «Боты и
живучесть онлайна» в [ROADMAP.md](ROADMAP.md).

**Планируемый клиент — Unity.** Текущий HTML/JS клиент
([templates/debug_map.html](templates/debug_map.html),
[static/map_editor.html](static/map_editor.html)) — это временный
инструмент для прототипирования и отладки сервера, не продакшен-клиент.
Не стоит вкладываться в его UI/UX сверх необходимого для отладки.

### Важно: расхождение сеттинга и кода

Проект несколько раз менял сеттинг (последняя явная итерация в коде —
подземелье/данж-кроулер с сундуками, комнатами и процедурной генерацией).
Игровое ядро (пошаговые бои, две команды, ходы, очки действия, оружие,
попадание/промах, overwatch) актуально и рабочее, но **терминология в коде
во многом не соответствует текущему сеттингу про роботов**:

- `Dungeon`/`DungeonMap` переименованы в `Arena`/`ArenaMap` (см. [ROADMAP.md](ROADMAP.md),
  Этап 1 п.7) — точечный ребрендинг сделан. `Room` — всё ещё наследие
  сеттинга "подземелье", но оставлен как внутреннее имя без потери смысла.
  Механику "выхода из данжа" убрали целиком; сундуки (`Chest`,
  `CELL_TYPE.CHEST`, `ActionType.OPEN_CHEST`) тоже убраны целиком
  (2026-07-14) — лут из них так и не переехал на реальную `Part` (Этап 2.5
  об этом писал), решили не тратить время на промежуточную ступень: лут
  теперь только по итогам матча (страж/Баррон, Этап 2 п.11, и пост-матчевая
  награда, Этап 3), в подземелье во время боя лута больше нет.
- `Player` — уже пилот с мехом: `Player.mech: Mech` собран из деталей по
  слотам `torso`/`legs`/`head` + **двух рук** `arms_left`/`arms_right`
  (см. [src/entities/mech.py](src/entities/mech.py)/[part.py](src/entities/part.py)),
  а итоговые `CharacterStats` — сумма характеристик деталей (`Mech.build_character_stats`).
  Только `action_points` на этой итерации остаётся "от пилота", не от детали.
  **Руки — одна деталь-выбор** (`arms_left`/`arms_right` обязаны быть одного
  типа, `Mech.check_slots`), но у каждой руки **раздельное здоровье** и **слот
  под одно оружие** (`Weapon.hand: "left"|"right"`, ROADMAP.md Этап 2 п.3-4):
  оружие в уничтоженной руке недоступно для атаки/овервотча (`Mech.arm_for`),
  статы рук (accuracy/melee_power) полные пока жива хотя бы одна рука. У игрока
  минимум одно оружие обязательно (`Player.check_weapon_loadout`), максимум по
  одному на руку. У деталей есть редкость (`Part.rarity`) и собственная
  прочность (locational damage: `Part.current_health`/`destroyed`,
  `Mech.apply_random_part_damage`/`recompute_live_stats`); лоадаут игроку
  выдаётся случайным пресетом из [src/mech_presets.py](src/mech_presets.py).
  `Enemy` мех пока не получил — остаётся на плоских `CharacterStats`/`Inventory`
  (осознанно, см. [ROADMAP.md](ROADMAP.md), Этап 1 п.3).
- Название пакета/репозитория (`dungeon_game`) тоже отражает старый сеттинг.
- Не удивляйся русским комментариям/сообщениям вперемешку с английскими
  именами — это осознанный стиль проекта (см. "Язык" ниже).

**Практическое следствие:** при переименовании/рефакторинге под новый
сеттинг — уточняй у пользователя, что нужно менять сразу, а что можно
оставить как внутреннее имя без потери смысла (`Actor`/`Turn`/`ActionHandler`
уже нейтральны и переименования не требуют). Не переименовывай сущности
"заодно" без запроса.

## Дорожная карта

Пошаговый план перехода от текущего боевого прототипа к играбельной версии
с мехами/деталями, а затем к БД/аккаунтам/прокачке — в [ROADMAP.md](ROADMAP.md).
Там же зафиксированы уже принятые с пользователем развилки (глубина системы
деталей, судьба PvE-наследия, стек для БД). Начиная работу по пункту плана —
сверься с ним, а закончив — отметь его там же (✅ + короткое "что именно
сделано / что осознанно не сделано"), как это сделано для пунктов про
`EXIT` и сундуки. Если по ходу работы замечаешь, что описание в этом файле
разошлось с кодом (как это уже было с `EXIT`/золотом до правки ниже) —
поправь AGENTS.md тут же, не откладывая.

Принципы проектирования дропа/наград/редкостей собраны в
[LOOT_DESIGN.md](LOOT_DESIGN.md) — при работе над любой системой лута
(страж, пост-матчевая награда, аффиксы, pity) сверяйся с ним; если
осознанно нарушаешь практику оттуда — зафиксируй почему.

## Архитектура

Backend на **FastAPI**, коммуникация с фронтом — **REST для лобби +
WebSocket для игры**. Хранение состояния — только в памяти процесса
(`LobbyManager.lobbies` и общий для всех лобби `LobbyManager.garages` —
обычные dict), базы данных нет.

### Основные модули

- [main.py](main.py) — FastAPI-приложение, HTTP-роуты лобби, debug dump/restore,
  единственный WS-эндпоинт `/ws/{lobby_id}/{player_id}`. Главный игровой
  цикл (обработка действия игрока → ход ИИ врагов → рассылка состояния)
  живёт прямо в WS-хендлере.
- [lobby.py](lobby.py) — `Lobby`: отдельно хранит подключённых
  `LobbyParticipant` (пилот → команда → actor ids), боевых `Player` и
  вебсокеты пилотов, создаёт `Game` при старте, реализует `GameObserver`
  для рассылки событий/состояния, проверяет владение актором, фильтрует
  видимое состояние по команде (`filter_visible_entities_for_team`) и
  доступные ходы по `owner_player_id` (`filter_available_moves`).
- [lobby_manager.py](lobby_manager.py) — реестр лобби в памяти.
- [src/game.py](src/game.py) — `Game`: жизненный цикл партии, порядок ходов,
  overwatch-триггеры при перемещении, условия окончания игры.
- [src/turn.py](src/turn.py) — `Turn`/`GamePhase`: фиксированная очередь
  боевых мехов с чередованием команд (`A1 → B1 → A2 → B2`, погибшие
  пропускаются) → `AI_ENEMY_PHASE` → следующий раунд с тем же порядком.
- [src/action.py](src/action.py) — `Action`/`ActionType`/`ActionResult` —
  контракт "что игрок/ИИ хочет сделать" и результат выполнения.
- [src/action_handler.py](src/action_handler.py) — вся игровая логика по
  типам действий: `MOVE`, `ATTACK`, `OVERWATCH`, `END_TURN`, `INSPECT`.
  Движение выполняется пошагово с проверкой overwatch на каждой клетке.
- [src/arena.py](src/arena.py) — `Arena`: карта боя + сущности на ней
  (враги, точки спавна). Поддерживает загрузку готовой карты
  (`_init_from_map`, сейчас используется в `lobby.py`) и процедурную
  генерацию комнат/коридоров (`_procedural_generate`, сейчас не используется
  при старте игры; код рабочий, но тестов на неё нет — не удаляй просто
  так, уточни у пользователя).
- [src/map.py](src/map.py) — `ArenaMap`: сетка тайлов, BFS-путь, доступные
  клетки для хода, линия видимости (алгоритм Брезенхема), can_see/can_shoot.
- [src/maps/default.py](src/maps/default.py) — захардкоженные готовые карты
  (`map_1`, `map_2` и т.д.), используются вместо процедурной генерации.
- [src/entities/](src/entities) — `base.py` (`Actor`, `CharacterStats`,
  `Weapon` с `Weapon.hand`, `Inventory`, `OverwatchState`), `player.py`
  (`Player` — боевой мех с отдельным `id`, `owner_player_id`, `loadout_id`
  и `mech: Mech`; включает `check_weapon_loadout`), `enemy.py`,
  `room.py`, `part.py` (`Part`, `PartSlot`, `PartRarity`, собственная
  прочность детали), `mech.py` (две руки `arms_left`/`arms_right` одного типа,
  `Mech.build_character_stats`/`arm_for`/`hand_side_of`; locational damage:
  `apply_random_part_damage`/`recompute_live_stats`).
- [src/parts_catalog.py](src/parts_catalog.py) — каталог деталей:
  стартовый набор `DEFAULT_*` (common, из него собирается нейтральный/
  тестовый `default_mech()`) и именные наборы трёх пресетов
  `STEELMAN_*`/`FIREWORKS_*`/`STRIKEFORCE_*` (rare); аналог
  [src/maps/default.py](src/maps/default.py), но для деталей меха.
  Генерация деталей с аффиксами — Этап 2 п.15 (см. [ROADMAP.md](ROADMAP.md)).
- [src/mech_presets.py](src/mech_presets.py) — `MechPreset` («мех + оружие
  под него» + лор-`description`): SteelMan (ближний бой), Fireworks Mk. 1
  (стрельба), StrikeForce (один рейлган). `get_random_mech_preset()` —
  текущий источник лоадаута игрока в `lobby.py`.
- [src/garage.py](src/garage.py) — общий in-memory профиль пилота:
  физические экземпляры деталей, метрики и два независимых `MechLoadout`.
  У каждого лоадаута свои установленные детали и фиксированное оружие
  стартового пресета; одна физическая деталь не может стоять на двух мехах.
  При старте матча оба лоадаута собираются в отдельных боевых акторов.
- [src/ai/](src/ai) — простой ИИ врагов (`SimpleEnemyAI`): дальняя атака →
  движение к ближайшему игроку → ближняя атака → overwatch → конец хода.
- [dto/](dto) — pydantic-модели HTTP/WS контрактов (отдельно от внутренних
  моделей `src/`, чтобы не отдавать клиенту лишнее сверх того, что уже
  прошло через фильтрацию видимости в `lobby.py`).
- [game_state_utils.py](game_state_utils.py) — сериализация/десериализация
  полного состояния игры для debug dump/restore эндпоинтов.
- [ws_utils.py](ws_utils.py) — коды закрытия WS-соединения.
- [ws_events.md](ws_events.md) — краткое описание событий WS сервер→клиент.

### Игровой цикл (пошагово)

1. Игрок создаёт лобби (`POST /lobbies`), другие подключаются
   (`POST /connect_lobby`), затем хост стартует игру (`POST /start_game`).
2. После старта вся коммуникация идёт через WS: клиент шлёт JSON-действие
   (`GameActionState`), сервер возвращает обновлённое состояние всем.
3. В целевом формате `1v1` пилотов каждый управляет двумя независимыми
   мехами. Ходы чередуются в стабильном порядке `A1 → B1 → A2 → B2`;
   после полного круга живых мехов идёт фаза ИИ-врагов, затем новый раунд
   снова начинает A1. Погибшие слоты пропускаются.
4. Каждому игроку сервер отправляет **своё** состояние: видимость сущностей
   ограничена командой (`can_see`, дистанция + линия видимости) и доступные
   клетки хода видны только текущему актору.
5. Действия: `MOVE`, `ATTACK`, `OVERWATCH` (засада — выстрел при движении
   врага в зоне видимости/дальности), `END_TURN`. Лута во время боя больше
   нет (сундуки убраны, 2026-07-14) — деталь можно получить только по
   итогам матча (страж/Баррон, ROADMAP.md Этап 2 п.11, и пост-матчевая
   награда, Этап 3); `Actor.trophies` — задел под этот пост-матчевый расчёт,
   в бою пока ничем не наполняется.
6. Попадание/промах считается через `Weapon.check_hit` на основе `accuracy`
   актора, `accuracy` оружия и дистанции (для дальнобойного — штраф за
   дистанцию). Урон одного попадания — `Weapon.roll_damage()` (разброс
   ±12.5%); при попадании по игроку урон дополнительно уходит в случайную
   деталь его меха (locational damage, см. `Mech.apply_random_part_damage`).

## Язык и стиль кода

- Комментарии, сообщения `GameEvent`, HTTP-описания (`description=`) —
  **на русском**. Имена классов/функций/переменных — на английском.
  Следуй этому же соглашению при добавлении кода.
- Форматирование — `black` (см. [Makefile](Makefile): `make code_style`).
- Модели данных — везде pydantic (v2). Внутренние игровые сущности
  (`src/`) и DTO для клиента (`dto/`) сознательно разделены — не смешивай их.

### Как быстро проверять изменения игровой логики

Домен ещё активно меняется (поля/сущности переименовываются или исчезают
целиком — так ушли `EXIT`, `Chest.gold`, а затем и весь `Chest`), поэтому
вкладываться в подробный
pytest-сьют, завязанный на точную форму DTO/моделей, невыгодно: он устареет
быстрее, чем окупится. Вместо этого — три уровня проверки, от дешёвого к
дорогому, использовать по умолчанию самый дешёвый из подходящих:

**Уровень 0 — прямой вызов игровой логики (использовать почти всегда).**
Без HTTP/WS/браузера: собрать `Arena`+`Player`(ы)+`Game` в памяти, руками
выставить `game.turn.current_actor`, вызвать
`await game.action_handler.perform_actor_action(actor, action)` и проверить
`ActionResult` + получившееся состояние (позиция/HP/инвентарь/`trophies`
и т.п.). Это на порядок быстрее полного цикла лобби/WS и устойчиво к
рефакторингу форматов, потому что проверяет поведенческий контракт, а не
форму сериализации. Шаблон (не тест на все случаи жизни, а форма, которую
адаптировать под конкретное действие):

```python
import asyncio, copy
from src.arena import Arena
from src.map import ArenaMap
from src.maps import default
from src.entities.player import Player
from src.entities.base import Inventory, Weapon
from src.action import Action, ActionType
from src.game import Game
from src.parts_catalog import default_mech

async def main():
    arena_map = ArenaMap(
        width=copy.deepcopy(default.map_2["width"]),
        height=copy.deepcopy(default.map_2["height"]),
        tiles=copy.deepcopy(default.map_2["tiles"]),
    )
    arena = Arena(enemies_num=1, map=arena_map)

    # статы собираются из меха (Mech.build_character_stats), не задаются руками;
    # у игрока обязательно хотя бы одно оружие, взятое в руку (Weapon.hand)
    mech = default_mech()
    player = Player(
        team=1,
        position=...,  # клетка рядом с тем, что проверяем
        mech=mech,
        stats=mech.build_character_stats(action_points=10),
        inventory=Inventory(weapons=[
            Weapon(type="melee", name="Клинок", damage=5, cost_ap=5,
                   range=1, accuracy=90, hand="right"),
        ]),
    )
    game = Game(arena=arena, players=[player])
    game.turn.current_actor = player
    player.current_action_points = 10

    action = Action(actor_id=str(player.id), type=ActionType.<ЧТО ПРОВЕРЯЕМ>, cell=...)
    result = await game.action_handler.perform_actor_action(player, action)

    assert result.performed, result.detail
    # дальше — проверка контракта поведения, а не точной формы данных, напр.:
    # assert player.stats.health == ...
    # assert enemy not in arena.enemies

asyncio.run(main())
```

Гонять как `python -c "..."` или временным файлом — коммитить не обязательно.
Если сценарий полезен и на будущее, оформи его рядом с
[test_serialization.py](test_serialization.py)/[balance_sim.py](balance_sim.py)
в корне репозитория — так уже сложился в проекте паттерн "рабочих
smoke-скриптов"; не заводи под каждую фичу новый файл в `tests/`, который
выглядит как формальный regression-тест, а по факту протухнет за один
рефакторинг модели.

На Windows в консоли кириллица в `print`/`assert` через Bash-обёртку может
отображаться битой из-за кодировки — сами данные при этом не повреждены;
если важно читать вывод глазами, добавь `PYTHONIOENCODING=utf-8` перед `python`.

**Уровень 1 — проводка лобби/HTTP/WS.** Использовать `fastapi.testclient.TestClient`
(как в [tests/test_lobby.py](tests/test_lobby.py)) только когда меняется сама
проводка — создание лобби, подключение, старт игры — а не игровая логика
внутри `Game`, для которой хватает уровня 0.

**Уровень 2 — браузер (самый дорогой, использовать реже всего).** Реальный
запуск сервера + [templates/debug_map.html](templates/debug_map.html) —
единственный способ проверить то, что нельзя проверить кодом: как это
выглядит и кликается (туман войны, CSS-классы, наличие/отсутствие кнопки).
Не использовать для проверки правил боя — это либо покрывает уровень 0,
либо требует вручную водить актора по карте через UI, что медленно и
чувствительно к конкретной карте (см. опыт с навигацией к сундуку по
`map_2` — быстрее оказалось проверить `ActionHandler` напрямую).

### Массовая симуляция для проверки баланса (`balance_sim.py`)

Три уровня выше — про корректность (действие сделало то, что должно). Для
другого вопроса — "а справедливо ли это по циферкам" (баланс пресетов
мехов/оружия) — есть отдельный инструмент,
[balance_sim.py](balance_sim.py), в корне репозитория, запускается как
`python balance_sim.py` (не pytest, тот же паттерн smoke-скрипта, что у
`test_serialization.py`). Появился как замена ручному
прогону 2v2 через 4 браузерные сессии, когда нужно быстро оценить, как
правка `Part`/`Weapon` в [src/parts_catalog.py](src/parts_catalog.py)/
[src/mech_presets.py](src/mech_presets.py) сдвигает баланс.

Что делает: чистое PvP 2v2 мехов (`Arena(enemies_num=0)`, без нейтрального
ИИ, чтобы не шуметь поверх баланса пресетов) на `map_2`: один пилот на
каждой стороне владеет двумя независимыми лоадаутами, обе команды водит
простой team-aware бот (`PvPBotAI`, адаптация `SimpleEnemyAI` из
[src/ai/enemy.py](src/ai/enemy.py) под PvP: стрельба → сближение → ближний
бой → овервотч → конец хода), напрямую через
`game.perform_actor_action` — без HTTP/WS/браузера, как в уровне 0, но
массово (по умолчанию `n=250` в `main()`): три зеркальных матч-апа 2v2,
три контр-пары 2v2 в **обеих** ориентациях команд и смешанный состав. Для
контр-пар скрипт выводит итог, усреднённый по ориентации команд, вместе с win
rate/средним числом раундов/остатком HP победителя.

Важные нюансы:
- В скрипте монки-патчится `src.action_handler.asyncio.sleep` (у
  `__perform_action_move` в проде намеренная `await asyncio.sleep(0.1)` на
  каждый шаг движения — анимация для реального клиента, см.
  [src/action_handler.py](src/action_handler.py)) — без этого 250×4 игр шли
  бы часами вместо секунд. Патч живёт только в `balance_sim.py`, продакшен-
  код не трогается. Новый целевой набор 2v2 содержит 10 серий по N игр,
  поэтому полный прогон занимает заметно дольше прежнего набора из четырёх.
- **Известный фоновый шум, не имеющий отношения к пресетам**: даже
  зеркальные матч-апы (идентичные составы с обеих сторон) дают заметный и
  зависящий от пресета перекос (в прогоне 2026-07-24 после перехода на
  `A1 → B1 → A2 → B2`: от 34/66 у Fireworks до 54/46 у StrikeForce).
  Порядок первой стороны намеренно стабилен по решению пользователя;
  рандомизацию/смену инициативы и её влияние на баланс рассматривать
  отдельной задачей после плейтеста с людьми — см. [ROADMAP.md](ROADMAP.md),
  Этап 2 п.13.
- Использовать при любой правке чисел в `parts_catalog.py`/`mech_presets.py`
  (HP, `speed`, `accuracy`, `melee_power`, урон/`cost_ap`/`range` оружия) —
  дешевле и быстрее, чем просить пользователя вручную гонять несколько
  реальных клиентов.

## На что обращать внимание при изменениях

- **Видимость и честность данных**: любое новое поле в состоянии игры,
  которое видно противнику, должно проходить через фильтрацию в
  `lobby.filter_visible_entities_for_team` / `filter_available_moves`,
  иначе появится читерская утечка информации (server-authoritative модель:
  клиент не должен получать данные, которых не должен видеть).
- **`ACTIONS_ENDS_TURN`** в [src/game.py](src/game.py) — если добавляешь новый
  тип действия, завершающий ход, не забудь внести его в это множество.
- **`Action.decode_action_type`** — при добавлении нового `ActionType`
  клиент присылает его как строку по имени enum-значения.
- Debug dump/restore (`/debug/dump_game_state`, `/debug/restore_game_state`)
  — вспомогательные эндпоинты только для отладки, не часть игрового
  API для клиента.
- Не удаляй "мёртвый" на первый взгляд код (например, процедурную генерацию
  арены) без подтверждения — учитывая, что сеттинг и требования ещё
  меняются, часть кода может быть временно не задействована, но нужна позже.
