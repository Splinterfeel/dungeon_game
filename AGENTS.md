# AGENTS.md

Инструкции для агентов (Claude Code и др.), работающих в этом репозитории.

## Что это за проект

Сервер пошаговой онлайн-игры на WebSocket. Игроки объединяются в две команды и
сражаются друг с другом (PvP), плюс на карте есть нейтральные враги под
управлением простого ИИ (PvE-элемент).

**Целевой сеттинг:** боевые роботы с пилотами внутри — идейно близко к
Front Mission 3, но онлайн, с двумя командами игроков вместо одиночной
кампании. Пилоты прокачивают статы, открывают новые детали, собирают
робота и тонко настраивают его сборку. **Главный фокус проекта —
кастомизация робота/лоадаута**, а не глубина боевой системы самой по себе.

**Планируемый клиент — Unity.** Текущий HTML/JS клиент
([templates/debug_map.html](templates/debug_map.html),
[static/map_editor.html](static/map_editor.html)) — это временный
инструмент для прототипирования и отладки сервера, не продакшен-клиент.
Не стоит вкладываться в его UI/UX сверх необходимого для отладки.

### Важно: расхождение сеттинга и кода

Проект несколько раз менял сеттинг (последняя явная итерация в коде —
подземелье/данж-кроулер с сундуками, комнатами и процедурной генерацией).
Игровое ядро (пошаговые бои, две команды, ходы, очки действия, оружие,
попадание/промах, overwatch) актуально и рабочее, но **терминология в коде
во многом не соответствует текущему сеттингу про роботов**:

- `Dungeon`, `DungeonMap`, `Room`, `Chest`, `CELL_TYPE.CHEST` — наследие
  сеттинга "подземелье". По факту сейчас это просто "карта боя" (arena/map),
  а сундуки/золото сейчас не используются по назначению (см. ниже).
- `Player` / `Enemy` — по факту это пилот в роботе, но статы (`health`,
  `damage`, `speed`, `accuracy` и т.д.) не разбиты на "пилот" и "робот/детали" —
  кастомизации робота из деталей в коде пока нет.
- Название пакета/репозитория (`dungeon_game`) тоже отражает старый сеттинг.
- Не удивляйся русским комментариям/сообщениям вперемешку с английскими
  именами — это осознанный стиль проекта (см. "Язык" ниже).

**Практическое следствие:** при переименовании/рефакторинге под новый
сеттинг — уточняй у пользователя, что нужно менять сразу, а что можно
оставить как внутреннее имя без потери смысла (например, `Dungeon` →
`Arena`/`Battlefield`, но `Actor`/`Turn`/`ActionHandler` уже нейтральны и
переименования не требуют). Не переименовывай сущности "заодно" без запроса.

## Архитектура

Backend на **FastAPI**, коммуникация с фронтом — **REST для лобби +
WebSocket для игры**. Хранение состояния — только в памяти процесса
(`LobbyManager.lobbies` — обычный dict), базы данных нет.

### Основные модули

- [main.py](main.py) — FastAPI-приложение, HTTP-роуты лобби, debug dump/restore,
  единственный WS-эндпоинт `/ws/{lobby_id}/{player_id}`. Главный игровой
  цикл (обработка действия игрока → ход ИИ врагов → рассылка состояния)
  живёт прямо в WS-хендлере.
- [lobby.py](lobby.py) — `Lobby`: хранит подключённых игроков и вебсокеты,
  создаёт `Game` при старте, реализует `GameObserver` для рассылки событий/
  состояния, фильтрует видимое состояние по команде (`filter_visible_entities_for_team`)
  и по текущему актору (`filter_available_moves`) перед отправкой конкретному игроку.
- [lobby_manager.py](lobby_manager.py) — реестр лобби в памяти.
- [src/game.py](src/game.py) — `Game`: жизненный цикл партии, порядок ходов,
  overwatch-триггеры при перемещении, условия окончания игры.
- [src/turn.py](src/turn.py) — `Turn`/`GamePhase`: фазы `TEAM_1_PHASE` →
  `TEAM_2_PHASE` → `AI_ENEMY_PHASE` → следующий раунд.
- [src/action.py](src/action.py) — `Action`/`ActionType`/`ActionResult` —
  контракт "что игрок/ИИ хочет сделать" и результат выполнения.
- [src/action_handler.py](src/action_handler.py) — вся игровая логика по
  типам действий: `MOVE`, `ATTACK`, `OVERWATCH`, `END_TURN`, `EXIT`, `INSPECT`.
  Движение выполняется пошагово с проверкой overwatch на каждой клетке.
- [src/dungeon.py](src/dungeon.py) — `Dungeon`: карта боя + сущности на ней
  (враги, сундуки, точки спавна/выхода). Поддерживает загрузку готовой карты
  (`_init_from_map`, сейчас используется в `lobby.py`) и процедурную
  генерацию комнат/коридоров (`_procedural_generate`, сейчас не используется
  при старте игры, но код рабочий и есть тесты на неё нет — не удаляй просто
  так, уточни у пользователя).
- [src/map.py](src/map.py) — `DungeonMap`: сетка тайлов, BFS-путь, доступные
  клетки для хода, линия видимости (алгоритм Брезенхема), can_see/can_shoot.
- [src/maps/default.py](src/maps/default.py) — захардкоженные готовые карты
  (`map_1`, `map_2` и т.д.), используются вместо процедурной генерации.
- [src/entities/](src/entities) — `base.py` (`Actor`, `CharacterStats`,
  `Weapon`, `Inventory`, `OverwatchState`), `player.py`, `enemy.py`,
  `chest.py`, `room.py`.
- [src/ai/](src/ai) — простой ИИ врагов (`SimpleEnemyAI`): дальняя атака →
  движение к ближайшему игроку → ближняя атака → overwatch → конец хода.
- [dto/](dto) — pydantic-модели HTTP/WS контрактов (отдельно от внутренних
  моделей `src/`, чтобы не отдавать клиенту лишнее — например, `dto.state`
  не пробрасывает золото сундука).
- [game_state_utils.py](game_state_utils.py) — сериализация/десериализация
  полного состояния игры для debug dump/restore эндпоинтов.
- [ws_utils.py](ws_utils.py) — коды закрытия WS-соединения.
- [ws_events.md](ws_events.md) — краткое описание событий WS сервер→клиент.

### Игровой цикл (пошагово)

1. Игрок создаёт лобби (`POST /lobbies`), другие подключаются
   (`POST /connect_lobby`), затем хост стартует игру (`POST /start_game`).
2. После старта вся коммуникация идёт через WS: клиент шлёт JSON-действие
   (`GameActionState`), сервер возвращает обновлённое состояние всем.
3. Ходы идут по фазам: команда 1 → команда 2 → фаза ИИ-врагов → новый раунд.
   Внутри фазы игроков ходят по очереди (`actor_ids_passed_turn`).
4. Каждому игроку сервер отправляет **своё** состояние: видимость сущностей
   ограничена командой (`can_see`, дистанция + линия видимости) и доступные
   клетки хода видны только текущему актору.
5. Действия: `MOVE`, `ATTACK`, `OVERWATCH` (засада — выстрел при движении
   врага в зоне видимости/дальности), `END_TURN`, `EXIT` (только команда 1,
   только на клетке выхода, только с полными ОД — механика из "данж"-сеттинга,
   вероятно нуждается в переосмыслении под новый сеттинг).
6. Попадание/промах считается через `Weapon.check_hit` на основе `accuracy`
   актора, `accuracy` оружия и дистанции (для дальнобойного — штраф за
   дистанцию).

## Язык и стиль кода

- Комментарии, сообщения `GameEvent`, HTTP-описания (`description=`) —
  **на русском**. Имена классов/функций/переменных — на английском.
  Следуй этому же соглашению при добавлении кода.
- Форматирование — `black` (см. [Makefile](Makefile): `make code_style`).
- Модели данных — везде pydantic (v2). Внутренние игровые сущности
  (`src/`) и DTO для клиента (`dto/`) сознательно разделены — не смешивай их.

## Тесты и запуск

- Запуск сервера: `uvicorn main:app --reload` (или [run_server.cmd](run_server.cmd)).
- Тесты: `pytest`. Есть [tests/test_lobby.py](tests/test_lobby.py) (HTTP через
  `TestClient`, лобби), [test_serialization.py](test_serialization.py) и
  [test_dump_restore.py](test_dump_restore.py) (сериализация/восстановление
  состояния игры). Обрати внимание: `tests/test_lobby.py` сам поднимает
  отдельный `uvicorn`-процесс на порту 8000 — при параллельном запуске
  тестов это может конфликтовать с уже запущенным дев-сервером.
- Зависимости не зафиксированы в repo (нет requirements.txt/pyproject.toml) —
  ставились вручную в окружение. Если добавляешь новую зависимость,
  уточни у пользователя, нужно ли заводить файл зависимостей.

## На что обращать внимание при изменениях

- **Видимость и честность данных**: любое новое поле в состоянии игры,
  которое видно противнику, должно проходить через фильтрацию в
  `lobby.filter_visible_entities_for_team` / `filter_available_moves`,
  иначе появится читерская утечка информации (server-authoritative модель:
  клиент не должен получать данные, которых не должен видеть).
- **`ACTIONS_ENDS_TURN`** в [src/game.py](src/game.py) — если добавляешь новый
  тип действия, завершающий ход, не забудь внести его в это множество.
- **`Action.decode_action_type`** — при добавлении нового `ActionType`
  клиент присылает его как строку по имени enum-значения.
- Debug dump/restore (`/debug/dump_game_state`, `/debug/restore_game_state`)
  — вспомогательные эндпоинты только для отладки, не часть игрового
  API для клиента.
- Не удаляй "мёртвый" на первый взгляд код (например, процедурную генерацию
  подземелья) без подтверждения — учитывая, что сеттинг и требования ещё
  меняются, часть кода может быть временно не задействована, но нужна позже.
