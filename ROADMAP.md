# ROADMAP.md

Пошаговый план перехода от текущего боевого прототипа (данж-сеттинг,
без персистентности) к играбельному прототипу с мехами/деталями, а затем
к БД, аккаунтам и прокачке. Основа для планирования — [AGENTS.md](AGENTS.md).

Решения, зафиксированные с пользователем на момент написания плана:

- Система деталей на первом этапе — **минимальная** (корпус, 1-2 слота
  оружия, ноги), без FM3-глубины. Углубление — отдельный этап после того,
  как core уже играбелен.
- Механика "выход из данжа" — **не нужна**, убираем.
- Сундуки — **убраны совсем** (решение изменено 2026-07-14; изначально
  планировали оставить в переосмысленном виде, см. Этап 1 п.2 и Этап 2.5
  п.1 ниже — это тот самый fallback, который в итоге и выбрали). Дроп
  деталей на уровне во время боя не нужен — лут только по итогам матча:
  страж/Баррон (Этап 2 п.11) и пост-матчевая награда (Этап 3).
- Процедурная генерация уровней — **не приоритет**, курс на ручные карты
  через [static/map_editor.html](static/map_editor.html). Жертвуем
  процедурной генерацией, если она будет мешать рефакторингу.
- Золото — убираем как отдельную сущность, награда после боя = валюта
  + дроп деталей, начисляется на этапе БД (Этап 3).
- Стек для БД — **PostgreSQL + SQLAlchemy/SQLModel**.

Дополнено по итогам аудита роадмепа (2026-07-13, зафиксировано с пользователем):

- **Петля лута — главная гипотеза продукта, проверяется раньше всего.**
  «Дроп → гараж → следующий бой» должна быть проверена самым дешёвым
  способом (in-memory, см. новый Этап 2.5), а не после БД. Оставшиеся
  пункты Этапа 2 переупорядочены «от крючка» — см. порядок работ в начале
  Этапа 2.
- **Борьба с мёртвым онлайном — боты, а не надежда на живой матчмейкинг.**
  Два механизма: подлив ботов в матчи (модель PUBG) и асинхронное PvP
  против сборок других игроков под управлением бота (модель Backpack
  Battles). Для обоих нужен бот-ИИ достойного уровня — см. раздел
  «Боты и живучесть онлайна». Живое PvP остаётся как режим, но перестаёт
  быть единственным способом получить матч.
- **Скиллы остаются независимыми от деталей.** Идея «скилл выучивается с
  надетой детали» (как в Front Mission 3: дроп детали = потенциально новый
  скилл) рассмотрена и **отклонена** пользователем — скиллы принадлежат
  пилоту (дерево из Этапа 2 п.8 / [skills_tree.json](skills_tree.json)),
  лут и скиллы — раздельные системы. Не возвращаться к идее без
  отдельного запроса.

3. ✅ **Минимальная модель Pilot / Mech / Part.** — сделано (2026-07-08).
   Решено с пользователем:
   - **Слоты меха на эту итерацию:** `torso` (здоровье), `legs` (скорость),
     `arms` (точность + сила удара рукопашного), `head` (дальность
     видимости) — по одному слоту каждого типа. Два слота под руки
     (левая/правая) и правило суммирования их бонусов — сознательно
     отложено на Этап 2 (см. ниже).
   - **Оружие независимо от рук** на этой итерации: `Inventory.weapons`
     остаётся как есть, руки дают точность/силу удара всему носимому
     оружию, а не какому-то конкретному. Привязка оружия к слоту руки
     (кол-во оружия = кол-во рук) — тоже отложена на Этап 2.
   - **Сила удара рук добавляется к урону оружия ближнего боя**
     (`weapon.type == "melee"`): `итоговый_урон = weapon.damage + actor.stats.melee_power`.
     Дальнобойное оружие не затрагивается. Заодно это чинит то, что
     `CharacterStats.damage` был объявлен, но нигде не учитывался в бою —
     поле переименовано в `melee_power` и наконец применяется.
   - **`Player` не переименовываем в `Pilot`** сейчас — просто добавляем
     ему поле `mech: Mech` (+ заготовки `xp: int = 0`, `level: int = 1`
     под будущую прокачку). Переименование — по-прежнему часть п. 6
     "точечный ребрендинг", отдельной задачей.
   - `Enemy` на этой итерации `Mech` не получает, остаётся на плоских
     `CharacterStats`/`Inventory` (только `damage` → `melee_power` в
     местах создания в [src/arena.py](src/arena.py) (тогда ещё
     `src/dungeon.py`, переименован в п. 7), для единообразия с
     переименованным полем).

   Реализация:
   - [src/entities/part.py](src/entities/part.py) (новый): `PartSlot`
     (`TORSO`/`LEGS`/`ARMS`/`HEAD`) + `Part` (id, slot, name, и
     `health`/`speed`/`accuracy`/`melee_power`/`view_distance`, все `= 0`
     по умолчанию — деталь заполняет только то, что относится к её слоту
     по конвенции, модель этого не форсирует).
   - [src/entities/mech.py](src/entities/mech.py) (новый): `Mech`
     (`torso`/`legs`/`arms`/`head`: `Part`) + метод
     `build_character_stats(action_points: int) -> CharacterStats`,
     суммирующий соответствующие поля всех 4 деталей и добавляющий
     переданные очки действия (единственный стат, который на этой
     итерации остаётся "от пилота", а не от детали).
   - [src/entities/base.py](src/entities/base.py): `CharacterStats.damage`
     → `CharacterStats.melee_power`.
   - [src/action_handler.py](src/action_handler.py): в
     `__perform_action_attack` — бонус силы удара для melee-оружия (см. выше).
   - [src/entities/player.py](src/entities/player.py): добавить
     `mech: Mech`, `xp: int = 0`, `level: int = 1`. Без pydantic-магии
     авто-сборки `stats` из `mech` в валидаторе — рассматривали, но
     отказались: `stats` должен на этой итерации оставаться обычным явным
     полем, потому что restore игры из дампа ([game_state_utils.py](game_state_utils.py))
     передаёт уже "просевший" в бою `stats` (health меньше максимума), и
     авто-пересборка из `mech` при каждой валидации молча затёрла бы
     прогресс боя. Вместо этого — явный вызов
     `mech.build_character_stats(action_points=...)` в каждом месте, где
     создаётся новый `Player` "с нуля" (в отличие от восстановления из
     дампа, где `stats` передаётся как есть).

11. **Нейтральный ИИ-страж ценного лута (модель Baron/Roshan из MOBA).**
    **Отложено (2026-07-24) в блок необязательного обогащения боя.**
    После перехода на формат `1v1` пилотов с двумя мехами у каждого бой сам
    по себе стал заметно живее, поэтому страж больше не считается
    жизненно необходимым следующим шагом для MVP. Возвращаться к пункту только
    если по плейтесту окажется, что текущему матчу не хватает второго центра
    напряжения или пост-матчевого дропа.
    Зафиксировано с пользователем (2026-07-12). Это же — решение по «судьбе
    PvE-наследия»: нейтральные ИИ-враги перестают быть автономными
    агрессорами (бродят + агрят ближайшего) и становятся **объективом, на
    который действуют игроки**, а не агентом, который действует на игроков.
    Зачем именно так:
    - **Честность в симметричном PvP.** Автономный агр «на ближайшего» —
      по сути RNG/позиционная лотерея в пользу одной команды, чего в
      server-authoritative модели быть не должно (см. AGENTS.md про честность
      данных). Объектив симметричен: обе команды сами решают, лезть ли, и
      когда.
    - **Осмысленность вместо шума.** Страж стягивает обе команды к одной
      точке карты (ценная деталь под ним) → возникает органическая PvP-драка
      за объектив, а не разрозненные стычки с бродячим мобом.
    - **Кормит стержень игры.** Награда за стража — двойная (вариант «A»,
      выбран с пользователем 2026-07-12): внутриматчевый **бафф** команде +
      редкая **деталь** как мета-вишенка (см. «Экономика награды» ниже).
      Деталь идёт в тюнинг/кастомизацию — ИИ становится воротами к лучшему
      луту, а не отвлечением от неё.
    - **Страж — опциональная цель, а не условие победы.** Зафиксировано с
      пользователем (2026-07-12): победа в катке определяется **победой над
      вражеской командой**, а не убийством стража. Страж — это ставка:
      команда 1 может рискнуть и полезть на стража, а команда 2 — ударить по
      команде 1, пока та завязла на мобе, и команда 1 не выдержит натиск
      стража и врага одновременно. Именно эта опциональность и создаёт
      интересные решения по ходам (то, чего игре сейчас не хватает, чтобы
      быть игрой, а не чередой ходов) — но она же порождает «bait objective»
      проблему, разобранную в «Экономике награды».
    - **Это упрощение ИИ, а не усложнение.** Переиспользуем `Enemy` +
      `SimpleEnemyAI`, но **снимаем** с него «двигаться к ближайшему игроку»
      (см. [src/ai/](src/ai)): страж держит зону, включается только когда
      к нему вошли/ударили (leash/aggro-on-contact). Сундука для гейта
      больше нет (убраны целиком, 2026-07-14, см. Этап 1 п.2) — по смерти
      стража деталь начисляется напрямую в `Actor.trophies` по формуле ниже,
      без промежуточного шага "подойти и открыть".

    **Экономика награды (зафиксировано с пользователем 2026-07-12, самое
    тонкое место — читать внимательно перед реализацией).**
    - **Проблема «bait objective».** Раз победа в катке = добить вражескую
      команду, а редкая деталь — это **мета-награда** (полезна после боя, в
      кастомизации, но не в текущей катке), то голая формула «деталь за
      стража» делает нырок на стража чистым минусом по ресурсам здесь и
      сейчас: кто полез — тот потратил AP/HP и подставился, кто ждал — тот
      наказал. Итог «кто напал на стража, тот и проиграл», объектив мёртв.
      В MOBA Барон работает именно потому, что его награда — внутриматчевое
      усиление, помогающее выиграть *текущую* игру; поэтому у нас деталь
      **обязана** сопровождаться внутриматчевым баффом (вариант «A»), иначе
      экономика — чистый downside.
    - **Два разных правила для двух наград — намеренная асимметрия, и она
      несущая:**
      - **Бафф (внутриматчевый) — бинарный, достаётся команде, ДОБившей
        стража** (по last-hit / добивающему удару). Это делает бафф
        «крадущимся»: команда 2 не может просто отсидеться свежей и наказать
        завязшую команду 1 — если она это сделает, команда 1 добьёт стража,
        получит бафф и выиграет последующую стычку. Значит команда 2 тоже
        обязана лезть к стражу (добить/оспорить) → **обе команды тратят
        ресурсы и экспонированы, драка идёт честно у объектива**, а не
        «одни устали об моба, другие приехали свежими» (та самая гравитация
        объектива из «Осмысленности» выше).
      - **Деталь (мета) — вероятностная, доля шанса ∝ вкладу команды в урон
        по стражу.** Снайпить last-hit'ом бессмысленно: подсосать 10% HP и
        добить — значит забрать себе лишь ~10% шанса на деталь, а 90% шанса
        остаётся команде, которая реально снесла 90%. Награждается вложение
        и риск, а не микро-снайп добивания (у нас нет смайт-тайминга MOBA,
        поэтому last-hit-стил детали ощущался бы несправедливо).
    - **Почему асимметрия хороша (богатая вилка решений).** Разные правила
      создают настоящий выбор в момент «страже осталось ~20% HP»: **добить**
      (получить бафф, но подставиться под свежую вторую команду) или
      **отойти** (сохранить свою долю шанса на деталь по вкладу, но отдать
      бафф — и, возможно, всего стража — второй команде). Плюс это
      **наказывает фрилоадера**: «начесал урон и сбежал» сохраняет долю
      шанса на деталь, но бафф отдаёт добившему — халявы нет. Единое правило
      «всё по вкладу» эту вилку убило бы и ослабило давление на команду 2
      лезть в драку.
    - **Формула детали (однозначно, чтобы не читалось двусмысленно).**
      В момент смерти стража, для **каждого игрока** команды независимо:
      `P(деталь) = BASE * (доля_урона_его_команды_по_стражу)`, где
      `BASE ≈ 0.2` (≈ 1 из 5 при 100% вклада — ориентир, не финал).
      - Ролл **независимый на каждого игрока** (поэтому «каждому в команде»
        = у каждого свои ~20% при полном вкладе команды, а не одна коробка
        на всех; один может выбить деталь, другой — нет, красть друг у друга
        нельзя — это гача-разброс, а не внутрикомандный конфликт).
      - Если деталь выпала — её **редкость катается ОТДЕЛЬНЫМ роллом** (вот
        здесь «везение игрока»: у разных игроков могут выпасть разные детали
        разной редкости).
      - Команда с 0 урона по стражу → 0 шанса (халявы за неучастие нет).
      - Пример-проверка: команда A нанесла 90%, команда B добила (10%).
        Деталь: у игроков A по `0.2*0.9 = 0.18`, у игроков B по
        `0.2*0.1 = 0.02`. Бафф: у команды B (добила). Ровно то, чего мы
        хотим — деталь тем, кто вложился; бафф тому, кто рискнул добить.
    - **Реализация — трекинг вклада.** Стражу нужно копить накопленный урон
      по командам (`damage_by_team`, или по актору с последующей сверткой в
      команду) + помнить, кто нанёс добивающий удар. Дешёвое новое
      состояние на сущности стража. Не забыть: урон от овервотча и от
      добивания тоже идёт в вклад.
    - **Пометки на потом (НЕ балансить сейчас, только зафиксировать):**
      - `BASE ≈ 0.2` — это кран мета-прогрессии; быстро/медленно решается
        на Этапе 3 (когда вообще появится накопление деталей в БД), сейчас
        число — только ориентир.
      - **Победа в катке должна оставаться бóльшим краном деталей, чем
        страж**, иначе вернётся вырожденная стратегия «фармить стража,
        забить на победу». Деталь со стража остаётся у команды, даже если
        она проиграла катку (мета персистентна) — это ок как утешение,
        пока победа даёт больше.

    Ключевые решения по балансу (зафиксированы с пользователем 2026-07-12,
    сознательно **против** прямого копирования Барона):
    - **НЕ делаем «чем дольше матч, тем ценнее лут».** У Барона бафф растёт
      со временем и поощряет пассивность — а у нас уже есть овервотч и
      дальний бой, которые и так тянут к кемперству. Складывать три стимула
      к пассивности вместе — вредно. Ценность лута от длительности матча
      **не зависит**.
    - **Отложенный, но фиксированный и заранее известный спавн** (ориентир —
      3-4-й ход, точное число подобрать при балансировке). Инверсия таймера
      Барона: первые ходы — про позиционирование, дальше — гонка за
      объективом. Фиксированный ход (а не случайный) — ради симметричной
      честности: обе команды заранее знают, когда и где появится страж.
    - **Драку решает не «кто первый добежал», а «кто может позволить себе
      вскрыть стража и пережить ответку».** Важный нюанс: чистый забег
      («кто быстрее дошёл и взял») в симметричной игре вырождается в победу
      более быстрого меха без боя (напр. Fireworks, speed 7, просто
      добегает и уходит) — и тогда страж снова «не к месту». PvP создаёт то,
      что убийство стража **дорого** (трата AP/HP/ходов) и оставляет
      команду **потрёпанной** — вторая команда либо перехватывает объектив,
      либо наказывает ослабленных (модель «третьего тела» из Hunt: Showdown).
      Забег остаётся частью решения, но не единственным фактором.
    - **Позиционирование стража на карте — симметричное** (центр / зеркальные
      относительно спавнов точки), чтобы объектив не был coin-flip'ом в
      пользу той команды, что ближе. На ручных картах (Этап 1 п.6) это
      ответственность автора карты, не кода.
    - **Исход п.5 решён (2026-07-14): дроп только по итогам матча**, без
      подбора сундука в бою. Деталь по формуле выше падает в `trophies`
      вложившимся сразу по смерти стража (это уже не "во время боя" в
      смысле блуждающего лута — стража всё равно нужно сначала убить, а
      это само по себе боевое взаимодействие, а не подбор предмета с пола).
      Внутриматчевый бафф выдаётся сразу по добиванию.
    - Опциональный анти-стейлмейт-слой (подкрепления / закрывающаяся зона,
      если катка всё же затягивается — модель AI Director из L4D/Vermintide)
      — **сознательно не сюда**, отдельным опциональным пунктом позже, если
      отложенного спавна стража окажется мало против затягивания.
    - Проверять — уровнем 0 из AGENTS.md: собрать `Arena` со стражем,
      убедиться, что (а) до нужного хода стража нет / он пассивен,
      (б) страж не уходит с зоны к «ближайшему игроку», (в) бафф достаётся
      команде, нанёсшей добивающий удар (проверить обеими командами как
      добивающими), (г) доля шанса на деталь считается по вкладу:
      смоделировать 90%/10% урона и сверить `P(деталь)` у игроков обеих
      команд с формулой `BASE * доля_урона` (для детерминизма — задать

12. **Ограниченный боезапас дальнобойного оружия.** **Отложено (2026-07-24)
    в тот же блок необязательного обогащения боя, что и страж.** После
    появления двух мехов у пилота и мех-целикового тюнинга бой уже получил
    больше темпа и решений; добавлять боезапас имеет смысл только если
    плейтест подтвердит, что ranged/overwatch всё ещё слишком душат pacing,
    и эту проблему не хватает решать текущими системами.
    Зафиксировано с
    пользователем (2026-07-12). Три причины сразу, каждая сама по себе
    достаточна, но именно вместе они и оправдывают механику:
    - **Кастомизация (стержень проекта).** Боезапас — **стат оружия, стоящий
      веса** (расходует бюджет из п.9, тот же принцип, что и плотность брони
      в п.10): «сколько патронов взять» решается **в гараже между боями**,
      это компромисс сборки (больше патронов → тяжелее → меньше
      скорости/брони), а не боевой микроменеджмент. Аналог выбора топлива/
      резины в Gran Turismo — ровно та рамка, в которую проект целится.
    - **Средство против кемперства.** Овервотч и дальний бой уже тянут игру
      к пассивности (та же проблема, что разбиралась у стража, п.11, и у
      слайдеров, п.10). Конечный боезапас — прямой рычаг против этого:
      нельзя вечно сидеть в засаде на дальнобое — патроны тратятся и на
      обычную атаку, и **на выстрел по триггеру овервотча одинаково**
      (это не отдельная более мягкая версия «только овервотч жжёт
      патроны» — обсуждалась, но выбрана полная: боезапас должен влиять на
      стратегию всего матча, не только гасить конкретно кемперов).
    - **Асимметрия ranged/melee.** Ближний бой становится «надёжным, но
      требует сближения», дальний — «мощным, но конечным ресурсом». Это
      усиливает уже заложенную разницу пресетов Fireworks/SteelMan
      (Этап 1, п.8) сверх голых цифр статов: Fireworks — бёрст, пока есть
      патроны, SteelMan — выжидает, когда у дальнобойного противника
      кончится боезапас, и добивает в ближнем бою.
    - **Явно НЕ делаем перезарядку в течение боя** (зафиксировано с
      пользователем 2026-07-12) — пул патронов расходуется на весь матч
      без пополнения; кончились патроны — остаётся только ближний бой

      ограничение объёма боевой бухгалтерии: не вводим `ActionType.RELOAD`
      и связанную с ним трату AP/хода на перезарядку — это уже была бы
      глубина боевой системы, а не кастомизации, чего проект избегает.
    - **Только дальнобойное оружие.** Melee остаётся безлимитным — вопрос

      [src/game.py](src/game.py)) — раньше это были два независимых расчёта
      урона, важно было не забыть про оба (см. похожую заметку про боезапас
      в п.12 выше).
    - Бонус `melee_power` за ближний бой по-прежнему прибавляется поверх
      результата `roll_damage()` как фиксированное число, без своего
      разброса — рандомизируется только собственно `Weapon.damage`.
    - Известное следствие округления при малых значениях урона: для
      оружия с `damage` в районе 2 разброс ±12.5% (диапазон ~1.75-2.25)
      после округления почти всегда возвращает то же число — т.е.
      слабое оружие (например, "Аварийный клинок" Fireworks, damage=2)
      ощутимого разброса на глаз не даст. Не считаем багом: разброс задан
      как процент, а не абсолютное значение; если для слабого оружия
      разброс тоже должен быть заметен — потребуется отдельная правка
      (например, минимальный абсолютный разброс ±1 вне зависимости от
      процента), не делали её сейчас, т.к. не просили.
    - Баланс-тесты (`balance_sim.py`) сознательно не гонялись сразу после
      этой правки по просьбе пользователя — сначала разброс, потом баланс
      (актуально в первую очередь для новых чисел `StrikeForce`, см. выше).
15. ✅ **Генерация деталей с аффиксами по редкости.** — сделано MVP
    (2026-07-24). Вместо сложной системы с несколькими аффиксами и
    зависимостью числа аффиксов от `PartRarity` реализована первая,
    прямолинейная версия: одна базовая деталь из каталога + один ролл силы
    аффикса `+0/+1/+2/+3`.
    - Порядок ролла награды теперь такой: шанс награды за матч →
      редкость детали → конкретная базовая деталь → один weighted-roll
      силы аффикса. Шансы аффикса зафиксированы пользователем как
      `+0 = 50%`, `+1 = 30%`, `+2 = 15%`, `+3 = 5%` — **одним роллом**, а
      не каскадом независимых проверок.
    - Пул аффиксов фиксирован по слоту детали, без экзотики и отрицательных
      модификаторов: `torso -> health`, `legs -> speed`,
      `head -> view_distance`, `arms -> accuracy | melee_power`.
    - Сила аффикса хранится прямо на экземпляре детали
      (`affix_tier`/`affix_stat`/`affix_value`) и сразу изменяет числовой
      стат детали; отдельной «системы аффиксов» с сущностями/таблицами пока
      не вводили.
    - `+0`-награды по-прежнему стараются сначала открывать ещё не
      найденные базовые типы деталей, как и раньше. Аффиксные (`+1..+3`)
      дропы уже могут быть повторными копиями знакомой базы — это и есть
      источник дальнейшей вариативности лута после открытия каталога.
    - ✅ **Аддендум (2026-07-24): грубая балансировка через headless-симулятор.**
      `balance_sim.py` получил `--affix-suite`: зеркальные прогоны
      «аффиксная команда vs базовая команда» с усреднением по обеим
      сторонам первого хода. На первой итерации это уже помогло поймать,
      что плоский `health`-аффикс слишком сильно баффает хрупкие пресеты
      (`StrikeForce`/`Fireworks`), поэтому шкала торса сделана
      консервативной: `+1/+1/+2`, а не линейной `+2/+4/+6`.
    - `PartRarity` теперь получает хотя бы минимальную механику не только
      через сам пул деталей, но и через то, что одна и та же базовая rare-
      деталь может выпасть в более сильной аффиксной версии. Полную
      многоаффиксную/эпическую систему с уникальными свойствами не делали —
      осознанно оставлено на потом, если MVP покажет ценность.

## Отряды мехов и чередование инициативы (in-memory, до БД) ✅ сделано (2026-07-24)

## Боты и живучесть онлайна (сквозная линия, зафиксировано 2026-07-13)

Структурный риск проекта: онлайн-PvP от одного разработчика умирает по
сценарию «мало игроков → нет матчей → отзывы "dead game" → ещё меньше
игроков». Ответ — боты как полноправный источник матчей, в трёх слоях по
нарастанию сложности. Живое PvP остаётся, но перестаёт быть единственным
способом сыграть.

1. **Бот-ИИ достойного уровня (фундамент, делать первым).** Не
   `SimpleEnemyAI` и не жёсткая эвристика `PvPBotAI` из
   [balance_sim.py](balance_sim.py), а ИИ, которого **не стыдно подставить
   вместо живого игрока**: понимает свой лоадаут (melee-архетип сближается,
   ranged держит дистанцию, «мех одного выстрела» вроде StrikeForce
   караулит момент), пользуется овервотчем и линией видимости, отступает
   на низком HP, играет вокруг объектива (страж, Этап 2 п.11), не делает
   очевидно самоубийственных ходов. `PvPBotAI` — стартовая точка, целевая
   планка заметно выше. Побочные выгоды: реалистичнее прогоны
   `balance_sim.py` и проверка гипотезы Этапа 2 п.13 (человекоподобная
   игра может ослабить преимущество второй команды). Ревью ИИ из Этапа 2
   п.7 делается здесь же, не отдельной задачей.
2. **Подлив ботов в матчи (модель PUBG).** Зафиксировано с пользователем
   (2026-07-13) как основной план борьбы с пустым лобби: если матч не
   набрал живых игроков, недостающие слоты добираются ботами из п.1 на
   пресетных/правдоподобных сборках. Опционально тем же механизмом: бот
   подхватывает меха отключившегося игрока (стыкуется с Этапом 4 п.2,
   переподключение). БД не требует — можно делать до Этапа 3, как только
   готов п.1.
3. **Асинхронное PvP против сборок других игроков (модель Backpack
   Battles).** Понравившийся пользователю вариант (2026-07-13): противник —
   **снепшот реальной сборки другого игрока** (лоадаут/тюнинг из БД,
   Этап 3), которую водит бот из п.1. Противник всегда «настоящий» — чужие
   решения по кастомизации и есть контент, — но живого человека ждать не
   нужно, онлайн не обязателен вовсе. Для игры с фокусом на кастомизации
   это основной кандидат на «гарантированный матч в любой момент».
   Требует Этап 3 (персистентные сборки) и п.1. Детали (выбор противника,
   рейтинг, награды за асинхронные бои) — проектировать при подходе к
   пункту, не заранее.

## Этап 3 — БД и персистентность (PostgreSQL + SQLAlchemy/SQLModel)

Начинать только когда меха/детали уже играбельны и не меняются каждую
неделю — миграции по живой схеме дороже, чем по стабильной.

1. **Инфраструктура.** `docker-compose` с Postgres для локальной
   разработки, Alembic для миграций, SQLModel-модели (хорошо сочетаются с
   уже используемым pydantic-стилем DTO).
2. **Схема БД (минимум):**
   - `Account` — учётка (см. п. 4 про авторизацию).
   - `Pilot` — профиль пилота: `account_id`, `xp`, `level`, `currency`.
   - `Part` — каталог деталей (переезжает из `src/parts_catalog.py` в БД
     как справочник; сид через миграцию).
   - `PilotPartOwnership` — какие детали пилоту доступны/разблокированы
     (не все детали доступны всем сразу).
   - `MechLoadout` — сохранённая сборка: `pilot_id` + ссылки на детали по
     слотам, может быть несколько на пилота.
   - `MatchHistory` — результаты боёв: участники, победитель, полученный
     опыт/валюта/дроп, таймстамп.
   - `Skill` — каталог скиллов (переезжает из `src/skills_catalog.py`,
     Этап 2 п.8, в БД как справочник; сид через миграцию, по аналогии с
     `Part`).
   - `PilotSkillOwnership` — какие скиллы пилот уже открыл через прокачку
     (постоянное «дерево развития» пилота между боями).
3. **Начисление прогрессии.** По окончании `Game` (сейчас `Game.ended`
   в [src/game.py](src/game.py)) — расчёт XP/валюты/дропа и запись в БД.
   Это новый код на границе `Lobby`/`Game`, сама логика боя не меняется.
   - **Скиллы и уровни (продолжение Этапа 2, п.8).** Именно здесь появляется
     то, чего не было в прототипе: реальное начисление XP → рост `Pilot.level`,
     а на каждый новый уровень — **выбор одного скилла из предложенных
     (1 из 2)** по уже зафиксированному статичному дереву (см. Этап 2, п.8:
     уровень 1 без скилла, уровень 2 — «Точный выстрел»/«Усиленный удар»,
     уровень 3 — ветка зависит от выбора на уровне 2). Зафиксировано с
     пользователем (2026-07-10): **само меню выбора рисует Unity-клиент**,
     не HTML-дебаг. Сервер отвечает за данные и правила, а не за UI:
     - при апе уровня по дереву определяет доступные на выбор 2 узла
       (для уровня 3 — по ветке, выбранной на уровне 2, т.е. нужно
       учитывать предпосылку/родительский узел, а не только `level`);
     - принимает выбор игрока и фиксирует его в `PilotSkillOwnership`;
     - выбранные скиллы пилота при старте боя грузятся из БД в
       `Player.skills` (в прототипе Этапа 2 они были захардкожены) —
       дальше боевой хук прока из п.8 работает без изменений.
     - Эндпоинты под это добавляются в п.5 ниже (профиль/уровень/скиллы).
4. **Авторизация — предложение, не блокирует остальное.**
   Сейчас клиент сам генерирует `player_id` (UUID) без какой-либо проверки.
   Рекомендация: на первом шаге БД **не делать полноценный логин/пароль**,
   а завести `Account`, привязанный к тому же клиентскому UUID (по сути —
   анонимный профиль с постоянным ID). Полноценную авторизацию
   (email/pass или OAuth) — отдельным этапом, когда появится реальный
   Unity-клиент и вопрос "как не дать украсть чужой прогресс" станет
   насущным. Если для тебя это принципиально раньше — можно поднять
   приоритет, но по умолчанию предлагаю не блокировать этим Этап 3.
5. **Эндпоинты:** профиль пилота (статы/XP/уровень), список деталей
   пилота, CRUD лоадаутов, история матчей, а также скиллы пилота: открытые
   скиллы + доступный на текущем уровне выбор и приём выбранного скилла
   (см. п.3, «Скиллы и уровни»).
6. **`lobby.py`/`start_game`:** отряд игрока собирается из выбранных
   `MechLoadout` из БД вместо одного лоадаута из каталога в коде.
   Каталог деталей в коде (Этап 1, п. 4) на этом этапе выпиливается —
   переезжает в БД полностью.
7. **Психология выпадения — якоря удержания.** По итогам аудита
   (2026-07-13); требует персистентности, поэтому здесь, а не в Этапе 2.
   Чистый флэт-шанс дропа даёт сухие серии, которые ломают крючок. Три
   стандартных и дешёвых механизма (конкретные числа — при реализации):
   - **Pity-таймер** — гарантированный дроп rare+ раз в N боёв без
     редкого дропа (счётчик на пилота в БД).
   - **Дубли → крафт/валюта** — повторная деталь не мусор, а прогресс:
     разобрать на валюту или обменять N дублей на деталь редкостью выше.
   - **Коллекция/кодекс** — каталог всех деталей с силуэтами ненайденных
     («осталось 3 детали серии "Голем"») — самый дешёвый якорь из трёх.
   Согласуется с уже зафиксированным принципом «победа — больший кран
   деталей, чем страж» (Этап 2 п.11, «Пометки на потом»).

## Этап 4 — сессии, устойчивость, продакшен (после того как есть БД)

Не расписываю подробно, чтобы не планировать слишком далеко вперёд —
детали будут яснее после Этапа 3. Крупными мазками:

1. Токены сессии вместо доверия к клиентскому `player_id` — нужно решить
   вместе с вопросом авторизации из Этапа 3.
2. Переподключение/устойчивость к разрыву соединения — уже частично есть
   (`websocket_endpoint` шлёт текущее состояние при реконнекте); можно
   опереться на существующий dump/restore
   ([game_state_utils.py](game_state_utils.py)) для восстановления после
   падения процесса, а не только реконнекта клиента.
3. Матчмейкинг вместо ручного шаринга лобби — опционально, по необходимости.
   Если к этому моменту решится открытый вопрос про score силы пилота/сборки
   (см. Этап 2, п.8, «Открытый вопрос... про шанс прока скиллов») — этот же
   score естественно использовать и здесь, для подбора соперников
   сопоставимой силы. Стыкуется с разделом «Боты и живучесть онлайна»:
   пустой матчмейкинг закрывается подливом ботов (п.2 раздела), а
   асинхронное PvP (п.3 раздела) снижает саму потребность в живом
   матчмейкинге.
4. Деплой/инфраструктура — отдельное обсуждение, когда дойдём до этой точки.

## Что намеренно не планируем сейчас

- Полноценный UI сборки меха в HTML-клиенте — не тратим время, целевой
  клиент Unity.
- Процедурную генерацию карт — не развиваем, но и не обязаны прямо сейчас
  удалять, если не мешает.
- Полноценную авторизацию — до тех пор, пока анонимного постоянного ID
  достаточно для тестирования прогрессии.
