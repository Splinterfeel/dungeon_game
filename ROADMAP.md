# ROADMAP.md

Пошаговый план перехода от текущего боевого прототипа (данж-сеттинг,
без персистентности) к играбельному прототипу с мехами/деталями, а затем
к БД, аккаунтам и прокачке. Основа для планирования — [AGENTS.md](AGENTS.md).

Решения, зафиксированные с пользователем на момент написания плана:

- Система деталей на первом этапе — **минимальная** (корпус, 1-2 слота
  оружия, ноги), без FM3-глубины. Углубление — отдельный этап после того,
  как core уже играбелен.
- Механика "выход из данжа" — **не нужна**, убираем.
- Сундуки — **хотим оставить** в переосмысленном виде: дроп деталей
  (особенно редких) прямо в бою. Если не впишется по геймплею/балансу —
  fallback на случайный дроп после победы в бою (без потери кода, см. 1.3).
- Процедурная генерация уровней — **не приоритет**, курс на ручные карты
  через [static/map_editor.html](static/map_editor.html). Жертвуем
  процедурной генерацией, если она будет мешать рефакторингу.
- Золото — убираем как отдельную сущность, награда после боя = валюта
  + дроп деталей, начисляется на этапе БД (Этап 3).
- Стек для БД — **PostgreSQL + SQLAlchemy/SQLModel**.

## Этап 0 — сделано

- [AGENTS.md](AGENTS.md): описан сеттинг, зафиксировано расхождение
  кода с сеттингом.

## Этап 1 — минимальный меха/детали core (без БД, всё ещё in-memory)

Цель этапа: убрать то, что явно не нужно от данж-сеттинга, и завести
минимальную, но настоящую систему "пилот + меха из деталей" вместо плоских
`CharacterStats`, которые сейчас руками прописаны в [lobby.py](lobby.py).
БД пока не трогаем — лоадаут при старте боя всё ещё выдаётся из
захардкоженного каталога в коде.

1. ✅ **Убрать механику выхода из данжа.** — сделано.
   - Удалены `ActionType.EXIT`, `ActionHandler.__perform_action_exit`,
     `CELL_TYPE.EXIT`, `Dungeon.exits`/`_generate_exits`, спец-случай для
     клетки выхода в `__perform_action_move`, поле `exits` в `dto/state.py`,
     кнопка/рендер/хендлер в [templates/debug_map.html](templates/debug_map.html)
     и палитра в [static/map_editor.html](static/map_editor.html).
     Клетки бывших выходов на `map_1`/`map_2` стали обычным полом.
   - Условие конца игры не зависело от "выхода" и не менялось
     (`Game.check_game_end` считает только смерти команд/врагов).

2. ✅ **Переосмыслить сундуки под дроп деталей.** — сделано (первая версия).
   - `Chest.gold: int` → `Chest.loot: str`. Пока это **заглушка**
     (`PLACEHOLDER_LOOT` в [src/entities/chest.py](src/entities/chest.py)) —
     случайное строковое имя вместо ссылки на реальную деталь, т.к. каталога
     деталей (п. 3) ещё нет. Когда он появится — `loot` меняется на ссылку
     на `Part`/`Weapon` с редкостью, остальная механика не трогается.
   - Реализован `ActionHandler.__perform_action_open_chest` +
     `Dungeon.remove_chest`: подбор возможен только на расстоянии ≤1 клетки
     (Chebyshev) от сундука; сундук исчезает с карты, лут уходит в новое
     поле `Actor.trophies: list[str]` — это и есть тот самый "reward pool"
     на актора, задел под будущую систему наград. Если механика подбора
     в бою не приживётся на плейтесте — переключить источник трофеев на
     пост-матчевую награду можно, не трогая `trophies` и сам расчёт наград.
   - `dto/state.py`: `ChestState` по-прежнему не отдаёт содержимое сундука
     клиенту заранее (раньше — `gold`, теперь — `loot`); `Actor.trophies`
     тоже пока не прокинуто в DTO — это чисто серверная бухгалтерия до
     появления реальной системы наград.
   - Не сделано намеренно: у трофея нет отображения в HTML-клиенте и нет
     влияния на баланс/выдачу конкретных деталей — это тема этапа 2.

3. **Минимальная модель Pilot / Mech / Part.**
   - Новый модуль, например `src/entities/mech.py` + `src/entities/part.py`:
     - `Part` (pydantic): `slot: Literal["chassis","legs","weapon"]`,
       модификаторы статов (`health`, `speed`, `accuracy`, `armor`?), для
       слота `weapon` — ссылка/встроенный `Weapon`.
     - `Mech`: `chassis: Part`, `legs: Part`, `weapons: list[Part]` (1-2 слота).
     - `Pilot` — по сути текущий `Player`, слегка расширенный: держит
       `mech: Mech` вместо голого `Inventory`/`CharacterStats`, плюс
       заготовка под будущее (`xp: int = 0`, `level: int = 1` — поля не
       используются до Этапа 3, но заводим сразу, чтобы не мигрировать
       модель дважды).
   - Функция сборки эффективных `CharacterStats` = база пилота + сумма
     модификаторов деталей меха. `Inventory.weapons` для игрока теперь
     собирается из `mech.weapons`, а не хардкодится.
   - `Enemy` пока не обязательно переводить на `Mech` — можно оставить
     плоские статы NPC-типа (турели/дикие роботы), если по сеттингу это
     оправдано; уточнить при реализации, не блокирует остальное.

4. **Каталог деталей (пока в коде, не в БД).**
   - `src/parts_catalog.py` (по аналогии с [src/maps/default.py](src/maps/default.py)):
     несколько готовых деталей на слот + 2-3 пресетных меха для быстрого
     старта без UI сборки.
   - `lobby.py`: `connect_player`/`start_game` берут пресетный `Mech` из
     каталога вместо инлайновых `Weapon(...)`.

5. **Ручные карты вместо процедурной генерации.**
   - Инвестируем в 2-3 карты в [src/maps/default.py](src/maps/default.py)
     через [static/map_editor.html](static/map_editor.html) (возможно,
     дополнить редактор кистью для клетки "сундук/дроп").
   - `Dungeon._procedural_generate` и связанный код (`Room`,
     `_generate_rooms/_generate_chests/_generate_enemies/_generate_exits`)
     не дорабатываем; удаляем по факту, когда/если он начнёт мешать
     переименованию `Dungeon` → `Arena`/`Battlefield` (см. п. 6). Не трогаем
     ради самого удаления — только если реально мешает.

6. **Точечный ребрендинг сеттинга (без фанатизма).**
   - `Dungeon`/`DungeonMap` → `Arena`/`BattleMap` (или похожее) —
     переименование крупное, но механическое; делать одним PR отдельно от
     логики деталей, чтобы дифф был читаемым.
   - `dungeon_game` как имя репозитория/пакета — обсудить отдельно, не
     блокирует остальную работу.

7. **HTML-клиент (только чтобы не потерять возможность отладки).**
   - Отрисовка HP/AP уже общая — должна продолжить работать при смене
     источника статов на "пилот+меха", если DTO (`dto/state.py`) не менять
     по составу полей.
   - Добавить в [templates/debug_map.html](templates/debug_map.html)
     минимальный вывод "из чего собран меха" (для отладки), полноценный
     билдер деталей в HTML не делаем — это забота будущего Unity-клиента.

**Критерий готовности этапа 1:** два игрока могут выбрать один из
пресетных мехов, зайти в лобби, сыграть бой на ручной карте, подобрать
дроп детали из сундука (или он просто падает по итогам боя — в
зависимости от того, как впишется), без единого упоминания "подземелья"
в игровых сообщениях.

## Этап 2 — углубление кастомизации (всё ещё без БД)

Делать только после того, как этап 1 пройден и опробован в игре — иначе
рискуем балансировать систему, которая ещё не проверена на прочность.

1. Больше слотов/деталей, редкости (common/rare/epic и т.п.), реальные
   компромиссы (например, тяжёлый корпус = меньше скорости).
2. Балансировка дроп-таблиц сундуков/наград — здесь и решится, остаётся
   ли подбор сундука в бою или переезжает целиком в пост-матчевую награду.
3. Тесты на сборку меха и расчёт итоговых статов (юнит-тесты, не только
   `tests/test_lobby.py` уровня HTTP).
4. Ревью `SimpleEnemyAI` — не потребует изменений логики принятия решений,
   но стоит проверить, что ИИ-противники с новыми статами/оружием всё ещё
   ведут себя разумно (дистанции атаки, точность и т.д.).

## Этап 3 — БД и персистентность (PostgreSQL + SQLAlchemy/SQLModel)

Начинать только когда меха/детали уже играбельны и не меняются каждую
неделю — миграции по живой схеме дороже, чем по стабильной.

1. **Инфраструктура.** `docker-compose` с Postgres для локальной
   разработки, Alembic для миграций, SQLModel-модели (хорошо сочетаются с
   уже используемым pydantic-стилем DTO).
2. **Схема БД (минимум):**
   - `Account` — учётка (см. п. 4 про авторизацию).
   - `Pilot` — профиль пилота: `account_id`, `xp`, `level`, `currency`.
   - `Part` — каталог деталей (переезжает из `src/parts_catalog.py` в БД
     как справочник; сид через миграцию).
   - `PilotPartOwnership` — какие детали пилоту доступны/разблокированы
     (не все детали доступны всем сразу).
   - `MechLoadout` — сохранённая сборка: `pilot_id` + ссылки на детали по
     слотам, может быть несколько на пилота.
   - `MatchHistory` — результаты боёв: участники, победитель, полученный
     опыт/валюта/дроп, таймстамп.
3. **Начисление прогрессии.** По окончании `Game` (сейчас `Game.ended`
   в [src/game.py](src/game.py)) — расчёт XP/валюты/дропа и запись в БД.
   Это новый код на границе `Lobby`/`Game`, сама логика боя не меняется.
4. **Авторизация — предложение, не блокирует остальное.**
   Сейчас клиент сам генерирует `player_id` (UUID) без какой-либо проверки.
   Рекомендация: на первом шаге БД **не делать полноценный логин/пароль**,
   а завести `Account`, привязанный к тому же клиентскому UUID (по сути —
   анонимный профиль с постоянным ID). Полноценную авторизацию
   (email/pass или OAuth) — отдельным этапом, когда появится реальный
   Unity-клиент и вопрос "как не дать украсть чужой прогресс" станет
   насущным. Если для тебя это принципиально раньше — можно поднять
   приоритет, но по умолчанию предлагаю не блокировать этим Этап 3.
5. **Эндпоинты:** профиль пилота (статы/XP/уровень), список деталей
   пилота, CRUD лоадаутов, история матчей.
6. **`lobby.py`/`start_game`:** лоадаут игрока берётся из БД
   (`MechLoadout` по умолчанию или выбранный) вместо каталога в коде.
   Каталог деталей в коде (Этап 1, п. 4) на этом этапе выпиливается —
   переезжает в БД полностью.

## Этап 4 — сессии, устойчивость, продакшен (после того как есть БД)

Не расписываю подробно, чтобы не планировать слишком далеко вперёд —
детали будут яснее после Этапа 3. Крупными мазками:

1. Токены сессии вместо доверия к клиентскому `player_id` — нужно решить
   вместе с вопросом авторизации из Этапа 3.
2. Переподключение/устойчивость к разрыву соединения — уже частично есть
   (`websocket_endpoint` шлёт текущее состояние при реконнекте); можно
   опереться на существующий dump/restore
   ([game_state_utils.py](game_state_utils.py)) для восстановления после
   падения процесса, а не только реконнекта клиента.
3. Матчмейкинг вместо ручного шаринга лобби — опционально, по необходимости.
4. Деплой/инфраструктура — отдельное обсуждение, когда дойдём до этой точки.

## Что намеренно не планируем сейчас

- Полноценный UI сборки меха в HTML-клиенте — не тратим время, целевой
  клиент Unity.
- Процедурную генерацию карт — не развиваем, но и не обязаны прямо сейчас
  удалять, если не мешает.
- Полноценную авторизацию — до тех пор, пока анонимного постоянного ID
  достаточно для тестирования прогрессии.
